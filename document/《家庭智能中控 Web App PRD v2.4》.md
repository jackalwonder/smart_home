二、术语定义  
  
1.	首页：家庭公共控制主入口页面。  
  
2.	主舞台：首页中央的 2.5D 户型展示与设备热点承载区域。  
  
3.	业务设备：后端对 HA 原始实体进行映射后形成的产品级设备对象。  
  
4.	原始实体：Home Assistant 中的原始 entity。  
  
5.	正式配置：已发布并对所有终端生效的正式版本配置。  
  
6.	草稿配置：编辑态中的临时修改结果，仅用于预览或等待发布。  
  
7.	Save All：设置中心对表单型配置的统一保存动作。  
  
8.	Publish：编辑态对首页主舞台结构型变更的正式发布动作。  
  
9.	控制成功：按本 PRD 定义的业务成功口径确认控制结果。  
  
10.	最终一致：后端与所有在线终端最终展示的状态一致。  
  
11.	墙板 / Kiosk 模式：公共常驻展示终端模式。  
  
12.	默认媒体设备：系统设置中手动指定的唯一首页音乐卡片来源设备。  
  
13.	通用动作模型：统一用于描述所有设备控制语义的 actionType + payload 模型。  
  
14.	只读设备：允许展示、告警、统计、摘要，但不进入控制执行链路的设备类型。  
  
15.	复杂设备：单一业务设备下可聚合多个原始实体、多个遥测项或多个控制项的设备。  
  
16.	轻展示：热点具备承载名称、摘要、状态、告警的能力，但不代表这些信息默认常驻完整展示在主舞台上。  
  
17.	实体角色：后端为业务设备下的原始实体标记的聚合角色，如主控、次控、遥测、状态、告警等。

三、产品定义  
  
3.1 产品定位  
  
家庭智能中控 Web App 是一个面向家庭共同使用的智能家居中控系统。产品以前端 Web App 为用户入口，以自有后端为统一业务中枢，以 Home Assistant 为底层设备接入平台，提供全屋总览、设备快控、统一配置、跨终端同步、首页编辑与家庭关键状态展示能力。  
  
3.2 项目背景  
  
当前家庭智能设备已接入 Home Assistant，但产品目标不再停留于前端直连 HA 的体验验证阶段，而是建设可持续演进的正式产品架构。为支撑配置持久化、多端同步、设备业务映射、外部数据接入、异常治理和未来能力扩展，需由后端统一承接业务能力和数据边界。  
  
3.3 产品目标  
  
1.	建立适合 iPad 墙板的家庭智能控制主入口。  
  
2.	通过空间主舞台形成“看空间、点设备、管全屋”的核心心智。  
  
3.	将高频控制压缩到首页一步或两步内完成。  
  
4.	通过后端统一设备、配置、状态与数据源，提升稳定性与可维护性。  
  
5.	为后续自动化、AI 控家、多端协同和更多设备平台预留统一架构。  
  
3.4 设计原则  
  
空间优先。  
  
快控优先。  
  
家庭共用。  
  
配置显式。  
  
动作分层。  
  
编辑隔离。  
  
系统分层。  
  
统一真源。  
  
稳定优先。  
  
信息克制。

四、用户与终端  
  
4.1 目标用户  
  
当前核心用户为家庭两位成员。后续可扩展更多家庭成员，但当前阶段不建设复杂权限体系。  
  
4.2 用户特征  
  
用户高频使用、操作碎片化，以查看状态与开关控制为主，对误触、延迟、状态不一致敏感，希望界面直观稳定，不需要理解底层系统概念。  
  
4.3 终端范围  
  
主终端为 iPad 9 墙板。  
  
兼容终端为其他 iPad 与桌面浏览器。  
  
手机端不作为当前阶段重点。  
  
4.4 终端策略  
  
墙板为公共入口，强调常驻、远距可读和快速响应。  
  
桌面端承载补充查看与配置编辑。  
  
多终端共享同一套后端配置真源与设备状态。  
  
4.5 墙板运行策略  
  
白天默认常亮。  
  
夜间默认降亮，不完全熄屏。  
  
连续 3 分钟 无操作自动恢复到首页默认浏览态。  
  
自动恢复规则仅对墙板 / Kiosk 模式生效，桌面浏览器默认不启用。  
  
编辑态、PIN 验证态、发布中态不触发自动回首页。

七、系统架构与职责边界  
  
7.1 总体架构  
  
flowchart LR  
  
A["前端 Web App<br/>墙板 / 桌面"] --> B["自有后端 API + WebSocket"]  
  
B --> C["配置数据库"]  
  
B --> D["HA 适配层"]

B --> F["缓存 / 聚合 / 推送层"]  
  
D --> G["Home Assistant"]

7.2 边界原则  
  
前端不得直连 Home Assistant。  
  
所有设备状态、控制、配置、草稿、发布、电量、统计、音乐卡片数据统一由后端提供。  
  
后端数据库是配置与映射唯一真源。  
  
前端本地缓存不是业务真源，只用于加速与降级兜底。  
  
7.3 后端核心职责  
  
1.	鉴权与管理 PIN 会话。  
  
2.	设备同步、映射、状态聚合。  
  
3.	控制转发、幂等、统一重试。  
  
4.	配置保存、草稿管理、版本发布。  
  
5.	WebSocket 推送。  
  
6.	电量数据接入、缓存、刷新。  
  
7.	降级与异常治理。  
  
8.	默认媒体设备聚合。  
  
9.	多实体业务设备聚合。  
  
10.	实体角色判定与设备能力归并。  
  
7.4 HA 实时同步与补偿机制  
  
后端优先订阅 HA 事件流。  
  
事件流断开后自动重连。  
  
重连成功后执行一次全量补偿同步。  
  
持续异常时退化为短周期轮询。  
  
前端收到“实时同步受限”状态后，需明确展示系统受限但不误报成功。

8.2 业务设备模型原则  
  
所有 UI 展示、统计、摘要、低电量、离线、常用设备排序，统一以业务设备 Device 为最小单位。  
  
不得以 HA 原始实体数量直接作为产品展示和统计口径。  
  
8.3 名称双轨  
  
原始名称来自 HA。  
  
展示名称由中控系统维护。  
  
HA 改名只更新原始名称，不自动覆盖展示名称。

8.5 多实体聚合规则  
  
一个业务设备 Device 可关联 1..N 个原始实体 Entity。  
  
其中：  
  
1.	Device 是产品级对象  
  
2.	Entity 是平台级对象  
  
3.	一个热点 Hotspot 只能绑定一个 device_id  
  
4.	一个常用设备 FavoriteDevices 只能收藏一个 device_id  
  
8.6 实体角色  
  
为避免多实体设备失控，业务设备下的实体应支持角色标记。  
  
冻结级角色包括：  
  
1.	PRIMARY_CONTROL  
  
2.	SECONDARY_CONTROL  
  
3.	TELEMETRY  
  
4.	STATUS  
  
5.	ALERT  
  
6.	DERIVED  
  
8.7 主控制对象  
  
defaultControlTarget 用于指定业务设备打开详情卡片时默认聚焦的主控制区域，而非等同于唯一控制实体。  
  
例如复杂设备可默认聚焦到某一子区域、主模式区或主要信息区。  
  
8.8 兼容原则  
  
现有灯光、空调、窗帘、音乐仍按原有产品体验运行。  
  
新增多实体聚合规则不要求前端直接理解 Home Assistant 原始实体细节，前端始终以 device_id + capabilities + runtimeState 工作。

十、首页  
  
10.1 首页定位  
  
首页承担空间总览、主设备快控、关键摘要展示、浮层分发和常用设备快速访问职责。  
  
10.2 首页结构  
  
1.	空间主舞台。  
  
2.	右侧信息栏。  
  
3.	快捷入口区。  
  
4.	底部电量统计条。  
  
5.	首页浮层层级。  
  
6.	编辑入口。  
  
10.3 布局预算  
  
主舞台建议不低于主要可视区域 60%。  
  
右侧栏宽度需设上限。  
  
底部只保留电量统计条一条强信息带。  
  
常用设备入口不得形成额外底部整块区域。  
  
10.4 首页显示规则  
  
每房间默认展示 2~3 个设备。  
  
高频快控设备优先占据主舞台高可达位置。  
  
详情型设备优先展示主入口对象。  
  
复杂设备默认以轻展示方式进入主舞台。  
  
超出部分进入弹窗或详情卡片。  
  
房间名称默认不显示，可配置为关闭、弱显示、仅编辑态显示。  
  
10.5 浏览态交互规则  
  
灯光点按直接进入快控链路。  
  
空调点按直接进入单设备卡片。  
  
窗帘点按直接进入单设备卡片。  
  
音乐卡片点击后打开音乐弹窗。  
  
复杂设备点击热点默认打开悬浮卡片或只读卡片。  
  
长按 400ms 仍保留为统一详情入口。  
  
首页必须提供显式详情入口。  
  
离线设备直接禁用。  
  
墙板模式下连续 3 分钟 无操作自动回首页默认态。  
  
10.6 复杂设备热点轻展示规则  
  
复杂设备的首页热点以图标化、低干扰、可识别为原则。  
  
“轻展示”表示热点具备承载设备名称、摘要、在线状态与关键告警的能力，但不代表这些信息默认长期完整显示在主舞台上。  
  
首页主舞台默认仅展示：  
  
1.	设备图标或设备类标识  
  
2.	必要的在线 / 离线状态提示  
  
3.	必要的异常 / 告警角标  
  
以下信息默认不常驻在主舞台热点旁：  
  
1.	完整设备名称  
  
2.	长摘要文案  
  
3.	多指标同时展开显示  
  
4.	可控项明细  
  
5.	多行状态文本  
  
上述信息仅在以下场景按需显示：  
  
1.	热点被点击或选中  
  
2.	打开悬浮卡片后  
  
3.	房间聚焦态或局部放大态  
  
4.	编辑态  
  
5.	右侧信息栏、房间摘要区或其他非主舞台承载区  
  
首页主舞台应优先保证空间结构可读性，不得因热点文案常驻导致户型主体被干扰。  
  
若主舞台与热点文案发生冲突，以减少热点文案常驻为优先处理原则。  
  
10.7 复杂设备点击规则  
  
对于复杂设备，点击热点默认打开悬浮卡片；若该设备被映射为只读设备，则打开只读卡片。  
  
复杂设备热点默认不直接执行复杂控制。  
  
复杂设备在主舞台上的默认职责是：  
  
1.	提供位置感知  
  
2.	提供轻状态提示  
  
3.	提供异常提醒入口  
  
4.	提供详情卡片入口  
  
复杂设备热点本体默认不承担以下职责：  
  
1.	长文案阅读  
  
2.	多参数同时编辑  
  
3.	多行状态长期展开  
  
4.	复杂控制项直接暴露  
  
若设备需要显示更多信息，应优先通过悬浮卡片展开，而非增加主舞台常驻内容。  
  
10.8 冰箱热点展示建议  
  
首页主舞台默认展示：  
  
1.	冰箱图标  
  
2.	必要状态角标：离线 / 异常 / 门未关  
  
以下信息不建议默认常驻在主舞台：  
  
1.	实时温度长文案  
  
2.	当前模式名称  
  
3.	多行状态文本  
  
4.	目标值与实时值同时展开  
  
点击热点后，在悬浮卡片中展示：  
  
1.	设备名称  
  
2.	关键实时指标  
  
3.	当前模式或主状态  
  
4.	异常 / 告警状态  
  
5.	可调节项或只读信息项  
  
如后续存在房间聚焦态或局部放大态，可在聚焦态下允许显示单行简化摘要。  
  
但该简化摘要不应作为默认浏览态常驻内容。  
  
10.9 传感器热点展示建议  
  
首页主舞台默认展示：  
  
1.	传感器图标  
  
2.	必要异常角标  
  
不建议默认常驻显示：  
  
1.	完整设备名称  
  
2.	长摘要  
  
3.	多指标并列文本  
  
4.	更新时间文案  
  
点击热点后，打开只读悬浮卡片，展示：  
  
1.	当前指标  
  
2.	其他环境或状态指标  
  
3.	最近更新时间  
  
4.	异常或告警信息  
  
传感器类设备默认不展示控制区。  
  
若后续存在房间聚焦态，可允许显示单行简化摘要，但仍不建议在默认浏览态下长期常驻。  
  
10.10 复杂设备当前阶段入口边界  
  
复杂设备当前阶段不单独形成首页固定分类快捷入口。  
  
复杂设备默认通过以下路径进入：  
  
1.	主舞台热点  
  
2.	右侧信息栏摘要区  
  
3.	详情卡片 / 悬浮卡片  
  
4.	后续设置中心或设备列表入口  
  
当前阶段首页固定快捷入口仍以灯光、空调、低电量、离线、常用设备为主。  
  
复杂设备是否出现在首页主舞台，由展示策略与设备能力共同决定。  
  
10.11 异常优先原则  
  
复杂设备若出现以下情况，可突破“仅图标展示”的默认规则，但仍应保持克制：  
  
1.	离线  
  
2.	严重异常  
  
3.	安全相关告警  
  
4.	持续未处理的重要提醒  
  
此时允许在热点附近增加轻量异常表达，例如：  
  
1.	异常角标  
  
2.	单词级短标签  
  
3.	高优先级颜色提醒  
  
但不建议展开为完整句式描述。  
  
异常优先于普通摘要，普通摘要优先通过悬浮卡片查看。

十一、设备与控制规则  
  
11.1 设备范围  
  
当前主闭环设备类型为：  
  
1.	灯光  
  
2.	空调  
  
3.	窗帘  
  
4.	音乐  
  
设备模型层需为后续复杂设备与多实体聚合设备预留统一扩展结构。  
  
未来扩展设备可包括但不限于：  
  
1.	冰箱  
  
2.	传感器  
  
3.	门锁  
  
4.	洗衣机 / 烘干机  
  
5.	空气净化器  
  
6.	新风系统  
  
7.	其他可映射为单业务设备 + 多实体的设备  
  
11.2 交互入口策略  
  
灯光在首页承担高频快控入口。  
  
空调在首页承担详情入口。  
  
窗帘在首页承担详情入口。  
  
音乐在首页承担媒体弹窗入口。  
  
复杂设备在首页承担悬浮卡片或只读卡片入口。  
  
本节仅定义产品交互入口，不定义设备特异性控制动作。  
  
所有控制动作统一由通用动作模型描述。  
  
11.3 控制请求责任  
  
前端只发起一次控制请求。  
  
所有设备控制统一采用通用动作模型 + 统一控制语义。  
  
后端统一负责：  
  
1.	幂等去重  
  
2.	唯一 requestId 校验  
  
3.	一次自动重试  
  
4.	动作语义校验  
  
5.	目标能力校验  
  
6.	最终状态确认  
  
7.	结果回推  
  
前端不得在控制超时前自行重发同一请求。  
  
前端交互策略与控制动作模型解耦。  
  
灯光、空调、窗帘、音乐、复杂设备的首页交互、详情入口、弹窗路径、成功口径与状态确认规则，继续按本 PRD 执行，不因动作模型统一而改变。  
  
11.3.1 统一动作模型  
  
所有控制请求必须使用统一 actionType。  
  
正式冻结版本中，不再使用设备类型专用动作枚举作为 PRD 正式契约。  
  
统一 actionType 冻结枚举如下：  
  
1.	TOGGLE_POWER  
  
2.	SET_POWER_STATE  
  
3.	SET_MODE  
  
4.	SET_VALUE  
  
5.	SET_TEMPERATURE  
  
6.	SET_POSITION  
  
7.	EXECUTE_ACTION  
  
所有控制请求统一通过 payload 表达具体控制目标。  
  
payload 最小结构包括：  
  
1.	targetScope：可选，表示控制作用域或子分区  
  
2.	targetKey：可选，表示控制字段或控制语义  
  
3.	value：可选，表示目标值  
  
4.	unit：可选，表示单位  
  
统一动作模型用于描述控制语义。  
  
设备展示方式、点按策略、是否进入详情卡片、是否支持快控，继续由设备能力与页面交互规则决定。  
  
11.3.2 值域与映射规则  
  
统一动作模型的值域、单位、模式映射表，以接口冻结文档为准，且必须统一跨设备类型复用。  
  
PRD 不再枚举灯光、空调、窗帘、音乐的设备特异性控制动作，也不再定义设备特异性控制映射。  
  
11.4 requestId 规则  
  
所有控制请求必须携带唯一 requestId。  
  
requestId 作用域为家庭级全局唯一。  
  
后端必须对相同 requestId 执行幂等控制。  
  
幂等窗口为 10 分钟。  
  
11.5 重复请求处理规则  
  
在幂等窗口内收到相同 requestId 时：  
  
1.	若原请求仍在处理中，返回处理中状态。  
  
2.	若原请求已成功，返回原成功结果。  
  
3.	若原请求已失败，返回原失败结果。  
  
4.	不得再次向 HA 下发同一控制请求。  
  
11.6 控制成功口径  
  
控制成功采用能力分层确认模式，不再按设备类型枚举控制动作。  
  
冻结级确认类型如下：  
  
1.	ACK_DRIVEN  
  
适用于以快速受理和状态校正为主的控制链路。  
  
受理成功起点：后端受理成功，且未判定为显式拒绝。  
  
最终控制成功仍以后续状态回推校正结果为准，不以受理成功直接等同于最终成功。  
  
状态机口径：前端先进入 ACK_SUCCESS，若短时间窗内收到目标状态回推则进入 SUCCESS；若状态未对齐，则进入 RECONCILING，最终可能进入 SUCCESS 或 STATE_MISMATCH / FAILED。  
  
2.	TARGET_STATE_DRIVEN  
  
适用于以目标状态达成为最终成功依据的控制链路。  
  
业务成功口径：目标状态在规定时间窗内由状态回推达成。  
  
仅接口受理成功但目标状态未达成，不视为最终成功。  
  
3.	PLAYBACK_STATE_DRIVEN  
  
适用于媒体或其他状态回推型控制链路。  
  
业务成功口径：后端受理成功且状态在规定时间窗内完成有效回推更新。  
  
最终展示以后端状态回推为准。  
  
设备属于哪一类确认类型，由设备能力与后端映射共同决定。  
  
前端不得在 PRD 层重新定义设备特异性动作语义。  
  
11.7 控制失败与超时口径  
  
显式失败包括：  
  
1.	后端鉴权失败  
  
2.	参数非法  
  
3.	设备离线  
  
4.	HA 接口显式返回失败  
  
5.	请求被后端拒绝  
  
6.	actionType 不受支持  
  
7.	targetKey / targetScope 与设备能力不匹配  
  
8.	设备为只读设备  
  
9.	数值超出允许范围  
  
超时失败包括：  
  
1.	ACK_DRIVEN 类型控制在规定时间窗内未完成有效校正  
  
2.	TARGET_STATE_DRIVEN 类型控制在规定时间窗内未收到目标状态回推  
  
3.	PLAYBACK_STATE_DRIVEN 类型控制在规定时间窗内未收到可接受的状态回推  
  
控制超时、状态不一致与显式失败，均应在统一状态字典中表达，不得因设备类型不同而建立第二套失败定义。  
  
11.8 自动重试规则  
  
自动重试仅由后端执行。  
  
每次请求最多重试一次。  
  
仅在非显式失败且具备瞬时异常特征时允许重试。  
  
离线设备、参数错误、权限错误、不支持的动作、不支持的目标、只读设备、数值越界均不重试。  
  
11.9 ACK 驱动类控制状态机  
  
ACK 驱动类控制链路使用以下状态：  
  
1.	PENDING  
  
2.	ACK_SUCCESS  
  
3.	SUCCESS  
  
4.	RECONCILING  
  
5.	FAILED  
  
6.	TIMEOUT  
  
7.	STATE_MISMATCH  
  
11.10 ACK 驱动类状态机图  
  
flowchart TB  
  
A["用户触发控制"] --> B["PENDING"]  
  
B --> C{"后端 / 适配层受理是否成功"}  
  
C -->|否| D["FAILED"]  
  
C -->|是| E["ACK_SUCCESS"]  
  
```
E --> F{"短时间窗内状态回推是否与目标一致"}
```

11.11 控制反馈流程  
  
前端发起一次控制请求 requestId 后，后端先执行幂等校验与能力校验。  
  
若设备离线、参数不合法、动作不支持、目标不支持或设备为只读设备，则直接失败。  
  
若校验通过，后端调用 Home Assistant 或对应后端适配层。  
  
控制结果统一进入以下链路之一：  
  
1.	ACK_DRIVEN  
  
2.	TARGET_STATE_DRIVEN  
  
3.	PLAYBACK_STATE_DRIVEN  
  
后端在必要时自动重试一次，并统一返回最终结果与状态推送。  
  
11.12 只读设备规则  
  
只读设备允许展示、告警、统计、摘要，但不进入控制执行链路。  
  
只读设备的相关失败不得进入正常控制执行态，而应在入口层阻断。  
  
点击只读设备时，应进入只读卡片或只读悬浮卡片，不得展示可执行控制区。  
  
11.13 离线设备规则  
  
离线设备在首页、详情卡片、分类弹窗、常用设备弹窗中均为禁用态。  
  
点击时仅提示“设备离线，暂不可控制”。  
  
离线设备不重试、不排队、不延迟执行。  
  
11.14 设备映射原则  
  
后端负责将 HA 实体统一映射为业务设备。  
  
支持自动识别和人工修正。  
  
房间归属、设备类型、默认控制对象、首页主设备均以业务设备配置为准。  
  
复杂设备允许多个实体聚合到单一业务设备，由后端完成聚合与角色判定。  
  
设备能力、控制确认类型、是否只读、是否复杂设备、是否支持快控，均以后端业务设备映射结果为准。

十二、右侧信息栏、快捷入口与音乐卡片  
  
12.1 右侧信息栏  
  
1.	时间日期。  
  
2.	天气。  
  
3.	音乐卡片。  
  
4.	全屋摘要。  
  
上述四类内容在产品聚合语义上统一归属于“右侧信息栏 / sidebar”，不再将其理解为首页总览接口中的四组平级聚合块。  
  
12.2 快捷入口  
  
固定入口：灯光、空调、低电量、离线。  
  
常用设备作为第 5 个入口或等价轻入口。  
  
12.3 音乐卡片边界  
  
首页只展示单一默认媒体设备。  
  
默认媒体设备由系统设置中手动指定。  
  
若未指定，则音乐卡片隐藏或显示未配置态。  
  
若已指定但设备离线，则卡片显示不可用态。  
  
“未指定”和“已指定但离线”是两个不同产品状态，前者属于未绑定，后者属于已绑定但不可用。  
  
第一阶段不支持多播放器自动切换。  
  
12.4 音乐卡片能力  
  
首页音乐卡片：

十四、电量统计  
  
14.1 接入方式  
  
后端统一接入外部电量服务。  
  
前端不得直接访问。  
  
后端负责绑定、刷新、缓存、异常标记和展示态返回。  
  
14.2 绑定与刷新语义  
  
“保存绑定”是独立配置动作。  
  
“手动刷新”是独立即时动作。  
  
二者都不属于 Save All。  
  
14.3 状态类型  
  
1.	未绑定。  
  
2.	已绑定加载中。  
  
3.	已绑定正常。  
  
4.	刷新失败。  
  
5.	缓存展示。  
  
6.	绑定失效。  
  
14.4 电量刷新 SLA  
  
1.	点击“手动刷新”后，前端应在 <= 200ms 内进入刷新中态。  
  
2.	正常外部数据源条件下，结果回显目标为 P95 <= 5s。  
  
3.	若超过 8s 仍无明确结果，系统应进入 FAILED 或 CACHE_STALE。  
  
4.	不得长时间停留在无反馈的刷新中态。  
  
14.5 异常出口  
  
外部数据源异常时：  
  
1.	若存在最近有效缓存，则进入 CACHE_STALE。  
  
2.	若无可用缓存，则进入 FAILED。  
  
3.	页面必须给出清晰异常提示，不得误报刷新成功。

十七、常用设备  
  
17.1 模块定位  
  
常用设备用于维护高频设备库，为首页常用设备快捷弹窗提供数据来源。  
  
17.2 设备范围  
  
当前默认范围为高频可控业务设备。  
  
默认纳入范围应优先覆盖首页无独立专属入口的高频可控设备。  
  
音乐类默认媒体设备因首页已有独立音乐卡片，当前阶段不作为常用设备默认范围。  
  
只读设备不作为常用设备默认范围。  
  
后续可根据设备能力策略扩展复杂设备，但不作为当前默认范围。  
  
17.3 功能要求  
  
支持勾选、取消勾选、显示当前数量、按房间筛选、按设备类型筛选。  
  
支持手动排序。  
  
排序结果应影响首页常用设备弹窗中的展示顺序。  
  
17.4 数量策略  
  
推荐数量 6~8 个。  
  
上限建议不超过 12 个。  
  
17.5 排序规则  
  
常用设备排序以用户手动排序结果为准。  
  
若用户未进行手动排序，则采用系统默认顺序。  
  
保存后排序结果写入后端数据库，并同步到所有在线终端。  
  
后续即使设备状态变化，也不应打乱已保存的手动顺序。

十八、编辑态工作台  
  
18.1 定位  
  
编辑态用于完成首页主舞台结构、热点坐标、布点、排序和显示结构调整。  
  
18.2 Publish 边界  
  
Publish 只用于以下内容：  
  
1.	热点新增与删除。  
  
2.	热点坐标。  
  
3.	舞台布点结构。  
  
4.	首页布局结构。  
  
5.	编辑态内排序与主舞台结构型调整。  
  
Publish 不负责：  
  
1.	设置中心表单型配置。  
  
2.	电量绑定。  
  
3.	阈值配置。  
  
4.	非舞台结构型系统配置。  
  
18.3 预览 / 发布模型  
  
预览只作用于当前编辑终端。  
  
未发布前，其他终端继续使用正式配置。  
  
发布成功后，所有在线终端同步切换到新正式版本。  
  
18.4 编辑锁机制  
  
后端采用 lease + heartbeat 草稿锁机制。  
  
锁租约时长：60s。  
  
前端续租心跳周期：20s。  
  
若未按期续租，锁自动释放。  
  
锁失效后，当前终端失去编辑权。  
  
18.5 强制接管规则  
  
当终端 B 强制接管后：  
  
1.	后端回收终端 A 的编辑锁。  
  
2.	向终端 A 推送“已失去编辑权”事件。  
  
3.	终端 A 自动降级为只读预览态。  
  
4.	终端 A 不得继续提交发布。  
  
18.6 坐标持久化规则  
  
所有热点和布点坐标必须以后端持久化的 0~1 归一化相对坐标为唯一真源。  
  
不得将绝对像素作为跨终端持久化格式。  
  
前端渲染时根据当前底图容器尺寸换算显示位置。

二十、接口与通信原则  
  
20.1 HTTP  
  
用于首屏拉取、配置获取、设备控制、Save All、Publish、电量绑定与刷新、默认媒体设备设置等。  
  
20.2 WebSocket  
  
用于设备状态变化、全屋摘要变化、配置版本更新、锁状态变化、失去编辑权通知、电量刷新结果、音乐状态推送。  
  
20.3 接口原则

Publish 请求必须带版本号。  
  
锁操作必须校验 lease 状态。  
  
错误码与状态枚举统一使用附录字典。  
  
读取类接口的缓存态默认作为成功响应中的运行状态表达，不作为默认失败外壳。

二十一、多端同步、降级与备份恢复  
  
21.1 多端同步  
  
控制完成后，所有在线终端状态最终一致。  
  
Save All 成功后，所有在线终端需收到 ==settings_updated== 并更新设置快照。  
  
发布成功后，所有在线终端切换到新正式版本。  
  
备份恢复成功后，所有在线终端需收到 ==backup_restore_completed== 并同时刷新设置与正式布局快照。  
  
预览草稿不得同步给未进入编辑态的终端。  
  
21.2 降级规则  
  
后端不可达时：  
  
1.	首页可显示缓存快照。  
  
2.	控制能力禁用。  
  
3.	设置中心只读或不可用。  
  
4.	编辑态禁止进入。  
  
5.	已在编辑态页面不得本地伪发布成功。  
  
HA 异常但后端正常时：  
  
1.	配置可查看。  
  
2.	控制受限。  
  
3.	首页可显示设备异常态。  
  
4.	电量模块不受影响。  
  
21.3 备份恢复  
  
至少支持手动备份与手动恢复。  
  
恢复后生成新正式版本。  
  
恢复成功后推送所有在线终端。  
  
恢复成功后的冻结级同步事件为 ==backup_restore_completed==，并应携带新的 ==settings_version==、==layout_version== 与 ==effective_at==。  
  
不将损坏草稿恢复为正式配置。

二十二、统计口径  
  
22.1 统计最小单位  
  
统一按业务设备统计，不按 HA 原始实体统计。  
  
22.2 在线 / 离线  
  
在线 / 离线数量按业务设备数统计。  
  
离线优先级高于低电量。  
  
22.3 低电量  
  
只有在线且具备电量能力的业务设备才参与低电量统计。  
  
离线设备不进入低电量集合。  
  
22.4 开启灯光数量  
  
按业务灯光设备中当前处于开启状态的数量统计。  
  
若业务灯光聚合多个实体，以主控制实体状态为准。  
  
22.5 运行设备数量  
  
运行摘要优先使用后端聚合后的业务运行状态。  
  
当业务运行状态可直接判断为运行中时，计入运行摘要。  
  
若业务运行状态缺失，则允许退化使用映射后的业务模式字段。  
  
具体字段映射以后端设备能力与接口冻结文档为准。  
  
22.6 位置类设备状态摘要  
  
位置 / 行程类设备按业务设备统计，统一采用三态口径：  
  
1.	已开  
  
2.	已关  
  
3.	部分开启  
  
首页只显示简化摘要，详情在对应详情卡片或弹窗中展开。  
  
22.7 复杂设备与只读设备统计补充  
  
复杂设备新增后，首页摘要、收藏、设备数量、在线离线数仍统一按业务设备统计，不按聚合后的实体数量统计。  
  
只读设备参与边界如下：  
  
1.	只读设备可参与设备数量与在线离线统计  
  
2.	只读设备默认不参与控制执行链路  
  
3.	只读设备是否参与首页热点展示由设备能力策略决定  
  
4.	只读设备是否参与常用设备由产品配置决定  
  
5.	只读设备可参与摘要与告警展示，但不进入控制执行态  
  
复杂设备的低电量边界如下：  
  
1.	只有业务设备本身具备业务级电量含义时，才参与低电量集合  
  
2.	内部某个电池实体不得自动导致整个复杂设备被计入低电量，除非映射规则明确声明该电池代表业务设备主电量

二十四、非功能需求  
  
24.1 性能  
  
首屏可交互、控制反馈、发布同步、重连恢复均需满足附录 D 指标。  
  
24.2 稳定性  
  
后端异常、HA 异常、电量异常必须解耦。  
  
单模块故障不得拖垮整页。  
  
缓存态与实时态必须明显区分。  
  
24.3 可维护性  
  
前端不直接感知 HA 原始复杂度。  
  
设备模型、配置模型、统计模型解耦。  
  
后端适配层与业务层分离。  
  
控制动作模型与前端交互策略解耦。  
  
复杂设备能力通过业务设备模型统一承载。  
  
PRD 不再为不同设备类型分别定义控制动作契约，所有正式控制语义统一由通用动作模型承载。  
  
24.4 安全性  
  
管理操作需要 PIN。  
  
敏感参数仅后端保存。  
  
PIN 仅后台可重置。  
  
关键操作建议写入最小审计日志。

二十六、验收标准  
  
26.1 架构闭环  
  
前端不直连 HA。  
  
所有业务数据统一由后端提供。  
  
后端完成设备控制、配置保存、发布、电量与推送闭环。  
  
26.2 控制闭环  
  
控制请求使用唯一 requestId 幂等。  
  
自动重试仅后端执行。  
  
离线设备直接禁用，不可控制。  
  
所有控制请求统一使用通用动作模型作为正式冻结口径。  
  
不得再以灯光、空调、窗帘、音乐的设备特异性控制动作作为正式 PRD 契约。  
  
设备交互入口与控制动作模型解耦，首页交互语义不因动作模型统一而改变。  
  
只读设备不进入控制执行链路。  
  
26.3 保存与发布边界  
  
Save All 只保存设置中心表单型配置。  
  
Publish 只发布编辑态主舞台结构型变更。  
  
两者不得复用同一生效路径。  
  
26.4 编辑锁  
  
后端采用 60s lease + 20s heartbeat。  
  
锁超时可自动释放。  
  
强制接管后原终端自动降级为只读预览态。  
  
失锁终端不得继续发布。  
  
26.5 坐标与适配  
  
所有热点坐标统一使用 0~1 相对坐标持久化。  
  
不同终端渲染不得出现明显错位。  
  
浏览态与编辑态位置换算规则一致。  
  
26.6 统计与摘要  
  
所有摘要统计统一按业务设备模型。  
  
离线优先于低电量。  
  
复杂设备仍按业务设备而非实体数统计。  
  
只读设备可参与展示与统计，但不参与控制执行。  
  
26.7 音乐卡片  
  
首页音乐卡片绑定单一默认媒体设备。  
  
音乐弹窗支持统一动作模型驱动的轻量媒体控制。  
  
第一阶段不支持多媒体切换。  
  
26.8 复杂设备展示  
  
复杂设备的首页热点默认采用图标化、低干扰展示。  
  
复杂设备默认通过悬浮卡片承载详情，不得在主舞台长期摊开高密度文案。  
  
异常信息优先于普通摘要。  
  
复杂设备当前阶段不单独形成首页固定分类快捷入口。  
  
26.9 动作模型冻结  
  
全部设备控制语义统一迁移到通用动作模型。  
  
不得再以设备专用动作枚举作为正式冻结契约。  
  
PRD 内部不再保留灯光、空调、窗帘、音乐的特异性控制动作定义与特异性控制映射。  
  
统一动作模型的值域、单位、模式映射表以后续接口冻结文档为准。  
  
26.10 SLA / SLO  
  
必须满足附录 D 的关键时延指标。  
  
⸻  
  
二十七、风险与应对  
  
27.1 主要风险  
  
1.	HA 命名不统一导致映射复杂。  
  
2.	电量数据源不稳定。  
  
3.	多端同步与锁机制实现复杂。  
  
4.	坐标适配处理不当导致错位。  
  
5.	音乐设备来源不明确导致 UI 不稳定。  
  
6.	墙板公共场景下管理入口滥用。  
  
7.	复杂设备实体聚合不当导致前后端语义不一致。  
  
8.	控制动作语义未完全统一导致联调歧义。  
  
27.2 应对策略  
  
1.	统一业务设备模型。  
  
2.	明确统计与状态口径。  
  
3.	使用 lease + heartbeat。  
  
4.	使用相对坐标真源。  
  
5.	明确默认媒体设备。  
  
6.	PIN 二次验证和后台重置。  
  
7.	由后端统一完成实体聚合与角色判定。  
  
8.	PRD 内部统一删除设备特异性控制契约，全部收口为通用动作模型。

二十八、结论  
  
《家庭智能中控 Web App PRD v2.4》在 v2.3.4 基础上，完成了冻结前跨文档一致性修订。  
  
本版本继续保持通用动作模型作为 PRD 唯一正式控制口径，同时补齐了 Save All 与备份恢复的同步事件边界，明确了默认媒体设备的绑定态 / 可用态语义分离，并收口了首页右侧信息栏的聚合语义。  
  
该版本可作为后续《接口清单 v2.4》和《统一响应体与错误体规范 v2.4》继续对齐的 PRD 冻结候选基线。

指标	目标值

目标状态达成类最终一致	P95 <= 4s  
  
位置 / 行程类最终一致	P95 <= 8s  
  
配置发布后多端切换	P95 <= 3s

“视觉反馈出现”指前端按压态、执行中态、禁用态反馈。

具体设备归属哪一类确认时延，以后续接口冻结文档和设备能力映射为准。  
  
其中“位置 / 行程类最终一致”属于 TARGET_STATE_DRIVEN 的长时运动子类指标，不构成独立的第四类控制确认模型。

附录 E：首页交互与状态规格说明（简版）  
  
E.1 浏览态  
  
首页默认进入浏览态。  
  
浏览态允许查看、快控、打开浮层。  
  
高频快控设备点按直接进入快控链路。  
  
详情型设备点按进入卡片。  
  
复杂设备点按进入悬浮卡片。  
  
只读设备点按进入只读卡片。  
  
离线设备不可控制。  
  
E.2 执行态  
  
触发控制后立即进入 PENDING。  
  
前端显示执行中样式。  
  
最终以业务成功口径或超时口径结束。  
  
E.3 ACK 驱动类成功态  
  
ACK 驱动类控制成功时：  
  
1.	先进入 ACK_SUCCESS  
  
2.	再等待状态回推  
  
3.	回推一致进入 SUCCESS  
  
4.	不一致进入 RECONCILING  
  
5.	校正失败进入 FAILED 或 STATE_MISMATCH  
  
E.4 目标状态达成类成功态  
  
目标状态达成类控制以目标状态回推为最终成功依据。  
  
仅接口成功但状态未达成时，不视为最终成功。  
  
E.5 媒体与其他状态回推类成功态  
  
状态回推类控制以规定时间窗内的有效状态回推为最终依据。  
  
最终展示以后端状态回推为准。  
  
E.6 只读设备态  
  
只读设备不进入控制执行态。  
  
点击后进入只读查看态。  
  
只读设备不得显示可执行控制区。  
  
E.7 失败态  
  
失败分为显式失败、超时失败、状态不一致。  
  
失败后应显示轻量错误提示，不得静默吞掉。  
  
E.8 编辑态  
  
编辑态与浏览态视觉强区分。  
  
编辑态禁用真实控制与所有主浮层。  
  
未发布变更仅当前端可预览。  
  
发布后全端更新。  
  
E.9 降级态  
  
后端不可达时进入受限态。  
  
受限态可看缓存，不可控制。  
  
设置中心只读或不可用。  
  
编辑态不可进入。
