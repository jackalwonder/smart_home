## 2. 项目概览与当前成熟度判断

项目是一个面向家庭智能中控的全栈应用，整体由 FastAPI 后端、React/Vite 前端、PostgreSQL、Redis、Home Assistant 集成、SGCC 电费侧车服务与 Docker Compose 编排组成。

当前成熟度判断：功能覆盖较完整，模块划分已经形成，但尚不建议进入冻结或上线状态。主要原因是：

- 后端测试套件当前失败，说明近期接口/构造器/DTO 演进与测试替身、契约消费方存在漂移。
- 认证密钥存在默认值且 Docker 运行路径未强制覆盖访问令牌密钥，这是上线级安全阻塞项。
- 设备控制链路可能把“未实际下发到 Home Assistant”的请求标记为成功，影响用户信任和联调判断。
- 实时事件链路具备基础 outbox/resume/ack 设计，但当前 ack 时机、窗口限制和持久 cursor 缺失使其更接近“尽力而为”，不应作为强一致状态来源。
- 前端核心页面和状态层偏大，业务解析大量依赖 `Record<string, unknown>` 与手写映射，类型契约保护弱。

综合评级：具备进入受控联调的基础，但需要先处理 P0/P1；不建议直接上线或冻结。

## 3. 项目结构与核心链路理解

### 3.1 后端结构

后端入口是 `backend/src/main.py`，通过 FastAPI 装配 API 路由、CORS、异常处理、静态资源、健康检查和 lifespan。模块以 `backend/src/modules/*` 组织，典型结构为 controller/service/repository/dto。

主要模块：

- `auth`：家庭、终端、访问令牌、PIN、bootstrap token、请求上下文解析。
- `home`：首页 overview 读取、查询模型组合。
- `devices` 与 `device_control`：设备查询、命令请求、幂等 key、HA 控制网关、控制投影。
- `settings`：系统设置、Home Assistant 连接、SGCC 绑定、媒体、备份、运行时控制。
- `energy`：电费/用电状态读取、刷新状态、手动刷新。
- `editor`：首页编辑器、草稿、发布、接管、差异预览、资源上传。
- `realtime`：WebSocket 网关、outbox 事件推送、ack、resume、快照 fallback。

依赖装配集中在 `backend/src/app/container.py`，以 `@lru_cache` 的 `get_container()` 暴露一个手写 service locator。DB 使用 async SQLAlchemy，迁移位于 `backend/alembic/versions`。

### 3.2 前端结构

前端入口是 `frontend/src/main.tsx`，路由在 `frontend/src/router.tsx`。技术栈为 React 19、TypeScript、Vite。

核心模块：

- `frontend/src/api/httpClient.ts`：统一 HTTP client、token 注入、错误归一。
- `frontend/src/store/appStore.ts`：自研 external store，集中保存 session、PIN、realtime、home、settings、editor、wsEvents。
- `frontend/src/auth/*`：bootstrap token、session、认证 API。
- `frontend/src/ws/wsClient.ts`：WebSocket 连接、重连、ack、lastEventId。
- `frontend/src/system/realtime.ts`：订阅实时事件后刷新 home/settings/editor snapshots。
- `frontend/src/view-models/*`：把后端 DTO 映射为页面消费模型。
- 核心页面：`HomeDashboardPage.tsx`、`SettingsWorkspacePage.tsx`、`DevicesWorkspacePage.tsx`、`EditorWorkbenchWorkspace.tsx`。

### 3.3 核心链路

认证链路：

1. 前端从 localStorage 读取 bootstrap token 或通过 terminal bootstrap/pairing 获取。
2. 后端 `AccessTokenResolver` 签发 HS256 JWT bearer token。
3. 前端 `httpClient` 注入 bearer token；WebSocket 通过 query 参数或协议传 token。
4. 后端 `RequestContextService` 解析 bearer、legacy token、bootstrap token、terminal/home 上下文。

首页链路：

1. 前端加载 `/api/v1/home/overview`。
2. 后端 home service 聚合设备、能源、媒体、系统连接、编辑器布局等读模型。
3. 前端 view-model 手写解析并落入 `appStore.home.overview`。

设备控制链路：

1. 前端提交设备控制请求。
2. 后端 `DeviceControlCommandService` 创建 request/action。
3. `HomeAssistantControlGateway` 根据 service/domain 生成 HA service call。
4. 写入控制状态与 outbox，前端通过轮询/实时事件刷新。

实时链路：

1. 后端模块写入 realtime outbox。
2. `RealtimeService` 管理连接、resume、推送、ack。
3. 前端 `RealtimeClient` 收事件并 ack，`system/realtime.ts` 根据事件类型刷新 snapshot。

SGCC/能源链路：

1. settings 模块维护 SGCC 绑定、登录二维码状态、运行时控制。
2. sidecar 服务位于 `services/sgcc_electricity_direct_qrcode`。
3. energy 模块读取用电、电费、刷新状态与来源更新时间。

## 4. 总体结论

本项目已经具备较完整的业务模块、数据库迁移、Docker Compose、CI、契约生成和 E2E smoke 基础，整体不是原型状态。但从联调/冻结角度看，目前存在三类关键问题：

- 安全基线不足：生产路径仍可能使用默认 JWT/加密/bootstrap secret，访问令牌安全边界不成立。
- 契约与验证不稳定：后端测试当前失败，DTO/服务构造器变更已经影响集成测试和单元测试。
- 关键业务链路语义不可靠：设备控制可能“空下发但成功”，实时事件 ack/resume 语义偏弱，SGCC 状态查询存在隐式写入。

建议：先完成 P0/P1 整改后再进入正式联调冻结。若必须联调，应限定为内网受控环境，禁用公网暴露，显式配置全部 secret，并把设备控制、SGCC、realtime 标为重点观测链路。

## 5. 关键问题清单（按 P0 / P1 / P2 / P3 分组）

### P0

#### P0-1 默认访问令牌与 bootstrap 密钥可进入运行环境

- 影响范围：认证、终端接入、家庭数据访问、所有 bearer token 保护接口。
- 发现依据：配置默认值与 Docker Compose 运行环境存在默认 secret。
- 证据位置：
  - `backend/src/shared/config/Settings.py:34-50`：`connection_encryption_secret`、`access_token_secret`、`bootstrap_token_secret` 均有固定默认值。
  - `docker-compose.yml:25`：`CONNECTION_ENCRYPTION_SECRET=${CONNECTION_ENCRYPTION_SECRET:-smart-home-local-secret}`。
  - `docker-compose.yml:30`：`BOOTSTRAP_TOKEN_SECRET=${BOOTSTRAP_TOKEN_SECRET:-smart-home-local-bootstrap-token-secret}`。
  - `docker-compose.yml` 未向 backend 注入 `ACCESS_TOKEN_SECRET`，应用会落回 `Settings.py` 默认值。
  - `backend/src/modules/auth/services/query/AccessTokenResolver.py:91-141`：使用 `access_token_secret` 发行 HS256 JWT。
  - `.env.example:15` 仅提示 `CONNECTION_ENCRYPTION_SECRET=change-this-in-real-env`，未覆盖 `ACCESS_TOKEN_SECRET` 与 `BOOTSTRAP_TOKEN_SECRET`。
- 为什么是问题：JWT 签名 secret 是认证边界；固定默认值意味着任何拿到代码或镜像配置的人都能构造有效 token。bootstrap token 默认 TTL 为 `2592000` 秒，进一步放大风险。
- 风险后果：未授权访问家庭设备、能源数据、设置接口；伪造终端或家庭上下文；已泄漏默认 secret 的部署无法通过普通 token 轮换完全补救。
- 修复建议：
  - 启动时强校验生产模式下 `ACCESS_TOKEN_SECRET`、`BOOTSTRAP_TOKEN_SECRET`、`CONNECTION_ENCRYPTION_SECRET` 必须显式配置且满足最小长度/熵要求。
  - `docker-compose.yml` 不应为安全 secret 提供可上线的默认值；仅在 `.env.example` 中保留占位说明。
  - 将 `.env.example` 补齐全部必需 secret，并明确“本地随机生成、生产不得复用”。
  - 增加配置单元测试，覆盖缺失/默认 secret 在 production profile 下启动失败。
- 是否已验证：已验证。通过静态代码与 Compose 配置确认。

#### P1-2 设备控制可能在未下发到 Home Assistant 时被标记为成功

- 影响范围：设备开关、窗帘、灯光、媒体等控制链路；前端控制状态；联调可信度。
- 发现依据：HA 网关在缺配置或不支持服务时直接 return，调用方随后标记成功。
- 证据位置：
  - `backend/src/infrastructure/ha/impl/HomeAssistantControlGateway.py:87-109`：无保存 HA 配置、缺 base_url/token、缺 bootstrap、或不支持 service_call 时直接 `return`。
  - `backend/src/modules/device_control/services/command/DeviceControlCommandService.py:358-420`：只有异常才进入 failed path。
  - `backend/src/modules/device_control/services/command/DeviceControlCommandService.py:422-469`：网关返回后将 action/request 标记为 `SUCCESS`，reason 为 `HA_ACKNOWLEDGED`。
- 为什么是问题：`return` 表达的是“未执行/不可执行”，但上层解释成“HA 已确认”。这是关键业务语义错误。
- 风险后果：用户看到控制成功但设备无变化；联调时误判 HA 已接收；幂等 key 可能阻止后续重试；控制审计记录失真。
- 修复建议：
  - 将 HA 网关返回值改为结构化结果：`sent/unsupported/misconfigured/skipped/failed`。
  - 对缺配置、缺 token、不支持 service call 明确抛出领域错误或返回失败状态。
  - `DeviceControlCommandService` 仅在 HA HTTP 调用成功且语义确认后标记 `SUCCESS`。
  - 增加测试覆盖：无 HA 配置、缺 token、不支持 domain/service、HA 4xx/5xx、超时。
- 是否已验证：部分验证。通过静态代码路径确认；未连接真实 HA 执行端到端验证。
- 未验证缺失条件：需要实际 HA 或 mock HTTP server 验证外部调用行为。

#### P1-3 SGCC 二维码状态查询接口存在隐式配置写入，且未要求管理 PIN

- 影响范围：SGCC 绑定、电费来源配置、设置页状态查询、审计归因。
- 发现依据：GET 状态接口会尝试自动绑定能源账号。
- 证据位置：
  - `backend/src/modules/settings/controllers/SettingsController.py:182-197`：`GET /api/v1/settings/sgcc-login-qrcode` 仅 `require_home=True`，未要求管理 PIN。
  - `backend/src/modules/settings/services/query/SgccLoginQrCodeService.py:167-181`：`get_status()` 调用 `_try_auto_bind_energy_account()`。
  - `backend/src/modules/settings/services/query/SgccLoginQrCodeService.py:288-339`：`_try_auto_bind_energy_account()` 在满足缓存/状态条件时执行 energy account upsert。
- 为什么是问题：GET 状态查询按 HTTP 语义应安全、幂等、无副作用；这里会修改能源绑定配置。设置类写操作通常应要求 PIN 或显式管理动作。
- 风险后果：页面轮询或自动刷新即可改变家庭能源绑定；审计显示不清楚是谁触发；缓存残留或 sidecar 状态异常时可能误绑定。
- 修复建议：
  - 将自动绑定改为显式 POST 操作，并要求管理 PIN/权限。
  - GET 只返回状态与建议动作，不执行 upsert。
  - 若保留自动绑定，至少增加一次性 nonce、状态机约束、审计 actor 与测试覆盖。
- 是否已验证：部分验证。静态确认调用链；未运行真实 SGCC sidecar 状态流。
- 未验证缺失条件：需要 SGCC sidecar 与 HA states/cache 数据验证触发条件。

### P2

#### P2-1 WebSocket access token 通过 query 参数传输，容易进入日志和历史记录

- 影响范围：实时连接、浏览器历史、代理/网关访问日志、服务端 access log。
- 发现依据：前端将 bearer token 拼入 `/ws?access_token=...`，后端 WebSocket 上下文允许 query token。
- 证据位置：
  - `frontend/src/ws/wsClient.ts:86-95`：构造 `access_token` query 参数。
  - `backend/src/modules/auth/services/query/RequestContextService.py:97-104`：解析 query/cookie token candidate。
  - `backend/src/modules/auth/services/query/RequestContextService.py:452-477`：WebSocket request 未禁用 query token。
  - `backend/src/modules/auth/services/query/RequestContextService.py:58-67`：已有 `_websocket_protocol_bearer` 支持，可作为迁移基础。
- 为什么是问题：query token 常被反向代理、浏览器调试工具、错误追踪和访问日志记录，泄漏概率高于 header/subprotocol。
- 风险后果：实时 token 泄漏后可被重放访问家庭实时事件；如果默认 secret 未修复，风险叠加。
- 修复建议：
  - 优先改为 WebSocket subprotocol bearer token 或短时一次性 WS ticket。
  - 保留 query token 作为短期兼容路径时，应默认关闭并记录弃用时间。
  - 限制 WS token TTL，服务端校验 origin。
- 是否已验证：已验证。通过静态代码确认。

#### P2-2 实时事件 ack/resume 语义偏弱，存在丢事件和状态过期窗口

- 影响范围：首页刷新、设置状态、编辑器协作、设备控制状态回填。
- 发现依据：客户端收到事件后立即 ack，服务端只保留内存 pending ack 与最近 100 条 resume 窗口。
- 证据位置：
  - `frontend/src/ws/wsClient.ts:122-130`：收到事件后先 `lastEventId = event.id` 并发送 ack，然后再 dedup/onEvent。
  - `backend/src/modules/realtime/services/RealtimeService.py:102`：pending push 查询最近 100 条。
  - `backend/src/modules/realtime/services/RealtimeService.py:106-107`：跳过 `created_at < connected_at` 的事件。
  - `backend/src/modules/realtime/services/RealtimeService.py:119-147`：resume backlog 最近 100 条，找不到 last id 才推 synthetic snapshot。
  - `backend/src/modules/realtime/services/RealtimeService.py:212-218`：ack 后标记 dispatched。
- 为什么是问题：ack 表示“已可靠处理”还是“已收到”没有明确边界。当前客户端在业务刷新成功前 ack；如果后续 snapshot 请求失败，服务端可能认为事件已派发完成。最近 100 条窗口对高频事件或长时间断线不够稳。
- 风险后果：UI 状态停留在旧值；编辑器/设置页错过关键刷新；排障困难，因为服务端 outbox 已标记 dispatched。
- 修复建议：
  - 将 ack 分层为 `received` 与 `processed`，或客户端仅在派生刷新成功后 ack。
  - 引入每 terminal/home 的持久 realtime cursor，而不是仅内存 pending ack。
  - resume 按 last_event_id 后分页拉取，超过窗口强制 snapshot 并向客户端暴露 resync reason。
  - 增加断线、重连、ack 后刷新失败、高频事件的集成测试。
- 是否已验证：部分验证。代码路径已确认；未做长连接压测和断网模拟。
- 未验证缺失条件：需要运行后端、前端、DB，并模拟网络抖动与事件洪峰。

#### P2-3 前端状态与 DTO 契约大量退化为 `Record<string, unknown>`，类型保护不足

- 影响范围：首页、设置页、view-model、接口契约演进。

#### P2-5 编辑器 diff 预览可能形成高频网络热点

- 影响范围：编辑器交互性能、后端 diff preview API、弱网体验。
- 发现依据：draft/base 变化后 250ms debounce 触发后端 diff 预览，无请求取消和更细粒度 dirty 判断。
- 证据位置：

- 为什么是问题：拖拽、批量调整、连续输入时可能不断触发 diff 计算；active flag 只能忽略过期响应，不能取消已发请求，也不能降低后端负载。
- 风险后果：编辑器卡顿；后端 CPU/DB 压力上升；多个客户端同时编辑时放大。

  - 对 draft 做 hash/dirty field 判断，无变化不请求。
  - 拖拽过程中暂停预览，交互结束后再请求。
  - 为 diff preview 增加后端限流或缓存。
- 是否已验证：部分验证。静态代码确认；未做性能压测。

#### P2-6 上传资源只校验 content-type 前缀，存在 SVG/伪造图片风险

- 发现依据：服务端依赖客户端提供的 `content_type` 与扩展名，未做真实文件头或 SVG 安全策略校验。
- 证据位置：

  - 白名单扩展名与 mime type，服务端重新确定 content type。
  - 增加恶意 SVG、伪造 mime、超大图片维度测试。

## 6. 架构与设计评估

### 6.1 正向评价

- 后端模块按业务域拆分，controllers/services/repositories 的基本边界已经形成。
- 认证、设备控制、实时 outbox、编辑器草稿、能源与设置模块均有独立模型和迁移，说明项目已经从页面原型进入业务系统阶段。
- 前后端有 OpenAPI 与 realtime schema 生成机制，CI 中也有契约生成检查意图。
- 设备控制引入 request/action、幂等 key、状态投影，比直接调用 HA 更适合做审计和恢复。
- 实时链路已经考虑 ack、resume、snapshot fallback，这是正确方向。

### 6.2 架构风险

- `backend/src/app/container.py` 是大型手写 service locator。优点是显式、可控；缺点是构造器变更容易造成测试替身和依赖装配漂移，当前 SGCC 测试失败已经暴露这一点。
- 前端 store 是全局 external store，状态域集中但类型较弱。它目前更像“全局 JSON 缓存”，而不是稳定的领域状态层。
- UI 层承担过多业务 orchestration，尤其 settings/editor。后续需求会倾向继续往大页面堆代码，重构成本会快速上升。
- Realtime 当前不应被视为可靠事件系统，只能作为刷新提示通道。关键状态仍应以 HTTP snapshot 为准，并在 UI/文档中明确。
- 设备控制链路需要把“请求已接收”“已下发 HA”“HA 成功响应”“设备状态已回填”分成不同状态，当前成功语义过粗。

### 6.3 模块边界建议

- 后端应保留 service 层作为业务规则入口，但需要为外部 gateway 定义明确结果类型，避免 `None/return` 被误解释。
- 前端应把 API DTO、domain model、view-model 分层固定下来，减少页面中的 unknown parsing。
- Settings 应拆为多个 feature slice；Editor 应拆出状态机和 side-effect hooks。
- Realtime 应作为 cross-cutting 基础设施，但不能直接替代查询模型一致性。
